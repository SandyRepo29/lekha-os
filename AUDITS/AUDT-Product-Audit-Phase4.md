## 9. Capability Redundancy Analysis

### Duplicate capabilities across modules

| Capability | Duplicate implementations | Recommendation |
|---|---|---|
| **AI Chat** | 20+ separate chat UIs across all modules — each independent, no history, no cross-module context | Consolidate into Governance Copilot™ (single chat, module-aware context injection). Keep per-module chat as a deep-link shortcut. |
| **Activity Timeline** | Vendor Hub™ (activity_feed via audit_logs) · Issue Hub™ (issue_history) · Auditor Collaboration™ (room activities) · Governance Agents™ (agent events) — 4 separate implementations | Single `activity_service.ts` consuming `audit_logs` for all entities. Module-specific views filter by entity type. |
| **AI Insights Cache** | `ai_compliance_insights` table used by: Evidence Vault™, Control Center™, Risk Lens™, Audit Management™, Trust Intelligence™, Contract Governance™, Regulatory Intelligence™ — all via the same table structure but duplicated service logic | Already centralised on one table. Consolidate `getCached()` / `saveCache()` into a single `ai-cache-service.ts`. |
| **Status Badge Components** | Each module defines its own badge components: `AuditStatusBadge`, `RiskStatusBadge`, `FindingStatusBadge`, `CapaStatusBadge`, `ControlStatusBadge`, `AssetStatusBadge`, `RegStatusBadge`, etc. | Create `StatusBadge` component accepting `status`, `variant`, and `colorMap` — all modules share one component. |
| **Stat Card (border-l-2 metric card)** | `AuditStat`, `ComplianceStat`, `RegStat`, `AssetStat`, `CcStat`, `AgentStat` — 6+ duplicates of the same component pattern | Single `ModuleStat` component. Already partially exists as `AuditStat` / `ComplianceStat` — merge into `components/ui/module-stat.tsx`. |
| **Sub-Nav Pill Pattern** | Each module (30 modules) has its own sub-nav implementation — some in `layout.tsx`, some as inline components | Single `SubNav` component accepting `items: { href, label }[]`. All 30 modules share one component. |
| **Notification / Alert** | `governance_alerts` (Trust Intelligence™) · in-app notification bell (M28) · email cron alerts — 3 separate notification channels with no common interface | Unified `notification-service.ts` that dispatches to: in-app bell, email (Resend), Slack (via Integration Hub™ webhook), webhook. |
| **Search / Filter Pattern** | Each module reimplements search + filter independently | `useModuleFilters(config)` hook + server-side `buildWhereClause(filters)` utility |
| **CSV Export** | 20+ separate CSV export routes with duplicated logic | `exportService.buildCSV(rows, columns)` + single `GET /api/v1/[module]/export` pattern |
| **Approval Patterns** | Policy lifecycle status advance · Issue exception approve/reject · Attestation sign-off · Auditor evidence accept/reject — 4 separate hard-coded approval patterns | Configurable `approval-service.ts` with entity type + approval chain + notification. |
| **Comments** | `external_comments` (Auditor Collaboration™) only | Single `comments` table with entity_type + entity_id polymorphic FK. All modules share one comment service. |
